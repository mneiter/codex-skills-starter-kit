# Description Optimization Patterns

## Purpose

Use this reference when improving an existing skill description without changing the skill's underlying workflow.

## Strong Description Traits

- intent-focused
- distinctive from neighboring skills
- concise
- explicit about the most relevant use cases
- clear about nearby cases that should not trigger

## Rewrite Patterns

### Weak To Strong Intent Framing

- weak: "Handles command execution and result analysis"
- stronger: "Use when a change needs repository validation before review or handoff"

### Trigger Phrase Expansion

Add a few adjacent phrases only when they reinforce the same intent.

- "review"
- "verify"
- "validate"
- "ready for handoff"

These terms help when they point to the same user goal rather than several different workflows.

### Negative Boundary Pattern

When overlap exists, add a short boundary:

- "Do not use this for broad architectural review"
- "Do not use this for implementation"
- "Do not use this for integration-level testing"

## What To Avoid

- long keyword lists
- examples that are too narrow to generalize
- descriptions that promise more than the skill body actually supports
- wording that hides the skill's real responsibility
